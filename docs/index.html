A governance layer for Claude Code

Auto-applies the safe edits. Surfaces the ones worth your review.

Hedwig watches every edit Claude Code makes and decides — per file, per session — which ones need your eyes. It adjusts from what happened last time.

Hedwig auto-applied an edit. You reverted it. The next similar edit gets surfaced. That's the loop.

The problem

Approval prompts are all or nothing.

ASK ON EVERYTHING

Too many prompts

You approve every trivial edit on reflex and stop reading the ones that actually matter.

ASK ON NOTHING

No visibility

Full auto-pilot until something goes wrong — including the security-sensitive file you'd have wanted to see first.

How it works

Score, decide, learn — locally, on every edit.

  1. Score

    Each edit gets scored on diff size, blast radius (how many files import this one), security sensitivity, and change pattern. Deterministic, model-independent.

  2. Decide

    Low-risk edits apply silently. Anything above the threshold surfaces with a plain reason: "security-sensitive file, no prior history — surfacing this one."

  3. Learn

    When an auto-applied edit gets reverted or fails verification, that becomes a corrective signal. The next similar edit gets a lower score. No synthetic data, no retraining — one gradient step per real outcome.

Why it's different

Most tools make you write the policy. Hedwig learns it.

01

Learns from outcomes

Every revert and verification failure is a training signal. The classifier updates once per event and generalizes to risk-similar edits across the repo.

02

Bidirectional oversight

Claude can self-declare low confidence on a risky edit and ask to pause. Hedwig honors that — oversight runs both ways, not just top-down.

03

Rules that match by meaning

Guidelines are retrieved semantically. "Avoid direct DB mutations" matches a task about raw SQL even if the words don't overlap.

Runs on your machine

Nothing leaves your machine.

The governance loop runs locally in Python. No cloud, no API key required.

No credentials
Risk scoring, the classifier, and the trace store run with no API key. Optional LLM features (semantic rule matching, pattern inference) use your own key when set.
Transparent
Run /hedwig-weights at any point to see exactly which signals have shifted and in which direction. The scoring is auditable, not opaque.
Per-repo, not per-person
Hedwig learns from this repo's history. Developer style isn't stable across sessions — the repo is the right unit of personalization.
