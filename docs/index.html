Governance layer for Claude Code

Auto-applies the safe edits. Surfaces the ones worth your review.

Hedwig watches every edit Claude Code makes and decides, per file and per session, which ones need your eyes. Learns from what you revert.

GitHub Docs Requirements

Hedwig auto-applied an edit. You reverted it. The next similar edit gets surfaced. That's the loop.

How it works

Score. Decide. Learn.

  1. Score the edit

    Diff size, blast radius, security sensitivity, change pattern. Deterministic. The model can't talk its way to a low score.

  2. Apply or surface

    Familiar low-risk edits go through silently. Anything riskier surfaces with a plain reason.

  3. Learn from outcomes

    A reverted edit or a failed verification is a training signal. One gradient step. No retraining, no synthetic data.

What it does

Most tools make you write the rules. Hedwig adapts to how you work.

Learned from outcomes

Every revert and verification failure updates the classifier. The next similar edit on any file in the repo scores lower.

Bidirectional

Claude can self-declare low confidence and ask to pause. Hedwig honors that. Tighten only, never loosen.

Pattern memory

Behavioral patterns accumulate evidence silently. Hedwig surfaces one question when it's confident. You confirm or decline.

Semantic rules

"Avoid direct DB mutations" matches a task about raw SQL. Rules are retrieved by meaning, not keyword overlap.

Memory is on disk.

Every decision, every regret event, the classifier weights — all persisted locally in SQLite. History carries across sessions. No API key, no cloud, nothing reset on restart.

Set ANTHROPIC_API_KEY to unlock semantic rule retrieval and LLM-powered pattern inference. Both opt-in.
