Smart Coder is implemented as a local Python CLI that sits between the developer and a Bedrock-hosted coding model. The model does not receive direct filesystem or shell authority. Instead, it communicates through a structured protocol with validated message types for read requests, intent declarations, check-ins, and candidate file updates. Smart Coder persists leases, approval history, autonomy preferences, plan revisions, and decision traces in a local SQLite database, which serves both as the runtime memory layer and as the primary artifact for later study analysis.

At runtime, every proposed action passes through a governed approval cascade. Hard constraints provide deterministic path-scoped read and write controls such as \texttt{always\_deny}, \texttt{always\_check\_in}, and \texttt{always\_allow}. Leases provide temporary, revocable capabilities derived from prior approvals, which also gives the design a capability-security interpretation~\cite{lampson1971protection}. Beyond these mechanisms, a heuristic policy engine scores actions using a weighted linear combination of interaction-history and risk features, with manually chosen priors over signals such as prior approvals and denials, review speed, edit distance, blast radius, change type, diff size, security sensitivity, and verification-failure history. Users do not manipulate raw thresholds directly; instead, they select qualitative autonomy modes such as \emph{strict}, \emph{balanced}, \emph{milestone}, and \emph{autonomous}.

Smart Coder organizes work into four enforced phases: \emph{research}, \emph{planning}, \emph{implementation}, and \emph{review}. Research emphasizes reading and note-taking, planning emphasizes intent declaration and plan approval, implementation permits code changes once the plan is accepted, and review emphasizes verification and surfacing failures. Phase gates prevent the agent from performing the wrong kind of action at the wrong time---for example, editing implementation files during planning without an explicit transition---and therefore act as a deterministic complement to the adaptive approval policy. The overall policy is hybrid rather than purely score-driven: some checkpoints are milestone-driven regardless of score, including plan approval, phase transitions, constrained files, deviations from an approved plan, and verification hotspots.

Smart Coder distinguishes between two initiators of oversight. The CLI may force a check-in when constraints, milestone triggers, or policy conditions indicate elevated risk. Separately, the model may proactively issue a check-in when it detects ambiguity, assumptions, or architectural tradeoffs. Because both are logged separately, Smart Coder supports a direct empirical question: which source of oversight is better calibrated under which conditions? Smart Coder also supports model-assisted rule authoring through \texttt{sc rules add}. A developer can express a new rule in freeform natural language, and the model compiles it into either (1) a deterministic path-scoped constraint suitable for CLI enforcement or (2) a behavioral guideline suitable for future prompt context. The CLI then validates the compiled result, previews it, and requires explicit confirmation before persistence.

The same trace store also drives prompt adaptation. Smart Coder summarizes prior interaction history into future system-prompt context, including qualitative trust summaries, behavioral guidelines, relevance-ranked historical corrections, and short logic notes describing what functionality changed in prior successful runs. In the current prototype, retrieval is lightweight and local: prior corrections, accepted guidelines, and logic notes are ranked against the current task and optional specification rather than injected purely by recency. This keeps the adaptation inspectable while moving beyond static rules files and purely model-internal memory~\cite{liang2026pahf}. Smart Coder also supports lightweight spec-aware planning through an optional \texttt{--spec} argument. The model is asked to map its plan to requirements and declare possible deviations, and the CLI treats missing requirement coverage or declared deviations as reasons to stop at the plan gate. After approval, updates are applied atomically and passed through a configured verification command, whose outcomes are logged and fed back into future autonomy decisions.

In practice, the user workflow is simple: the developer adds hard rules or softer guidance, supplies a task and optional specification, lets Smart Coder plan and read context, answers only the check-ins that matter, and then relies on later runs to adapt from prior approvals, corrections, and verification outcomes. This is the level at which the demo operates: not as a raw model session, but as a governed workflow whose behavior changes across interactions.
